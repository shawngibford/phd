# Lotka–Volterra Surrogate — PHD Harness Example

This example demonstrates the full PHD experiment loop on a classical SciML task: recovering
the parameters of a Lotka–Volterra (predator-prey) ODE from trajectory observations. It is
the canonical Slice 1 validation project — the gate that confirms parallel search, crash-safe
checkpointing, kill-and-resume, and KEPT rows flowing through to `/phd:write` all work
before heavier models are layered on.

---

## What the example does

The Lotka–Volterra system is:

    dx/dt =  α·x − β·x·y      (prey population)
    dy/dt = −γ·y + δ·x·y      (predator population)

True parameters (hidden from the model): `α=1.0, β=0.1, γ=1.5, δ=0.075`.

`project.jl` implements a surrogate-learning task: the "model" is a four-parameter vector
`[α̂, β̂, γ̂, δ̂]` that `train_step` nudges toward the true values via coordinate-wise
finite-difference gradient descent. Ground-truth trajectories are generated by a hand-rolled
RK4 integrator (no `DifferentialEquations.jl` dependency). The data is split 80/20 in time:
the training window is the first 80% of the trajectory; the held-out window (used by `score`)
is the last 20%. The score is `rel_l2` — the relative L2 error between the model's predicted
held-out trajectory and the ground truth.

Each epoch runs in well under 1 millisecond. The budget (`budget_s` in `job.json`) controls
the wall-clock cutoff; with `budget_s=120` and `max_epochs=200` the run is budget-bound only
on very slow hardware.

## Prerequisites

Julia 1.7+ is required (uses `Random.Xoshiro`; falls back to `MersenneTwister` on older
versions). No packages beyond the Julia standard library are needed for `project.jl` itself.
The harness (`runner.jl`, `metric.jl`, `device.jl`) requires only `Serialization` and
`Printf` from stdlib.

## Setting up a job directory

Copy and edit `job.example.json` for each hypothesis you want to test. The full field contract is:

```json
{
  "hid":          "h001",
  "diff":         "description of what changed vs the previous hypothesis",
  "budget_s":     120,
  "backend":      "cpu",
  "seed":         42,
  "max_epochs":   200,
  "parallel_k":   2,
  "project_file": "/absolute/path/to/phd/examples/lotka-volterra/project.jl"
}
```

Field notes:

- `hid` — matches the ledger row identifier (e.g., `h001`, `h002`). Used in `result.json` and log output.
- `diff` — human-readable description of the change being tested. Written verbatim into the ledger `change:` field.
- `budget_s` — wall-clock budget in seconds. The runner exits the epoch loop when this is exhausted.
- `backend` — recorded in `result.json`; has no effect on dispatch in v1 (CPU-only).
- `seed` — integer seed for all random number generation. Must be fixed per hypothesis for reproducibility.
- `max_epochs` — hard cap on epochs regardless of budget.
- `parallel_k` — number of concurrent experiments the supervisor may launch. Used by `supervisor.jl`; ignored by `runner.jl` directly.
- `project_file` — absolute path to this `project.jl`. Can also be set via the environment variable `RUNNER_PROJECT_FILE` instead.

## Running a single experiment

Create a job directory and write a `job.json` into it:

```bash
PHD_PLUGIN=/path/to/phd
LV_PROJECT="$PHD_PLUGIN/examples/lotka-volterra/project.jl"

mkdir -p my-research/runs/h001
cp "$PHD_PLUGIN/examples/lotka-volterra/job.example.json" my-research/runs/h001/job.json

# Edit the project_file path in job.json:
sed -i '' "s|/path/to/phd|$PHD_PLUGIN|" my-research/runs/h001/job.json
```

Run the experiment (foreground, for verification):

```bash
julia "$PHD_PLUGIN/harness/runner.jl" my-research/runs/h001
```

Or set the project file via environment variable (overrides `project_file` in `job.json`):

```bash
RUNNER_PROJECT_FILE="$LV_PROJECT" julia "$PHD_PLUGIN/harness/runner.jl" my-research/runs/h001
```

## Running as a detached job (daemon mode)

The full daemon workflow runs experiments as detached OS processes so the Claude session
can be closed and the run continues. To launch a single experiment detached:

```bash
PHD_PLUGIN=/path/to/phd
JOBDIR=my-research/runs/h001

nohup julia "$PHD_PLUGIN/harness/runner.jl" "$JOBDIR" \
    > "$JOBDIR/stdout.log" 2> "$JOBDIR/stderr.log" &

echo "Runner PID: $!"
```

Monitor progress by watching the heartbeat and epoch directory:

```bash
watch -n 2 'cat my-research/runs/h001/heartbeat; echo; ls my-research/runs/h001/epoch/'
```

Check final status:

```bash
cat my-research/runs/h001/status        # PENDING → RUNNING → DONE | FAILED
cat my-research/runs/h001/result.json   # final ExperimentResult
```

## Resuming a killed job

If a job is killed (OOM, power loss, manual `kill`), the last completed epoch is preserved
in `epoch/NNNN.ckpt`. Resume from that checkpoint:

```bash
julia "$PHD_PLUGIN/harness/runner.jl" my-research/runs/h001 --resume
```

The runner reads the highest-numbered `.ckpt` file, restores model parameters and RNG state,
and continues from the next epoch. No data is lost beyond the in-flight epoch at the time of
the kill.

## Running parallel experiments (K=2)

To run K=2 experiments concurrently (two hypotheses in parallel), create two job directories,
write a `job.json` into each, and launch them as separate detached processes:

```bash
for hid in h001 h002; do
    JOBDIR="my-research/runs/$hid"
    mkdir -p "$JOBDIR"
    cp "$PHD_PLUGIN/examples/lotka-volterra/job.example.json" "$JOBDIR/job.json"
    # Edit hid and seed in each job.json before launching.
    nohup julia "$PHD_PLUGIN/harness/runner.jl" "$JOBDIR" \
        > "$JOBDIR/stdout.log" 2> "$JOBDIR/stderr.log" &
    echo "Launched $hid (PID $!)"
done
```

The supervisor (`supervisor.jl`) and poller (`poller.jl`) manage this automatically when
`/phd:daemon start` is active — they read `parallel_k` from the job spec and enforce the cap.

## How results feed the gate

The Slice 1 validation gate confirms the end-to-end spine:

1. **Parallel search runs detached.** The two nohup jobs above run independently; killing one
   does not affect the other. Each writes its own `status`, `heartbeat`, and `epoch/` directory.

2. **A killed job resumes from the last `.ckpt`.** Kill any running job with `kill <PID>`,
   then re-run with `--resume`. Verify that `epoch/NNNN.ckpt` exists for each completed epoch
   and that the resumed run starts from epoch N+1.

3. **KEPT rows reach the `/phd:write` draft.** After a run completes:
   - The poller (`/phd:run --tick`) reads `result.json`, compares to the current best, and
     appends a `KEPT` or `DISCARDED` row to `LEDGER.md`.
   - Running `/phd:write` then reads only KEPT rows and drafts `paper/results.md`.
   - The resulting draft will contain one subsection per KEPT hypothesis, citing the run
     artifacts and metric values exactly as recorded in the ledger.

The gate passes when all three steps work end-to-end on this Lotka–Volterra project without
errors. Only then does Slice 2 (governor + backbone) begin.

## Expected output

A completed `runs/h001/` directory should look like:

```
runs/h001/
├── job.json
├── status          ← "DONE"
├── heartbeat       ← {"epoch":200,"ts":1750000000.0}
├── result.json     ← {"hid":"h001","score":0.042,"wall_s":3.1,"backend":"cpu","meta":{...}}
├── stdout.log
├── stderr.log
└── epoch/
    ├── 0001.ckpt
    ├── 0001.json   ← {"epoch":1,"loss":0.85,"score":0.91,"wall_s":0.0003}
    ├── 0002.ckpt
    ├── 0002.json
    ...
    └── 0200.ckpt
```

The `result.json` score is the **best** `rel_l2` on the held-out trajectory across all epochs.
A well-converged run should achieve `rel_l2 < 0.10` within 100–200 epochs from the perturbed
starting point, and approach `rel_l2 < 0.01` as the parameter estimates converge to the true
values.
